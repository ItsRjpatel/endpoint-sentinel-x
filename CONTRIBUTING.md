# Contributing Guidelines

Thank you for contributing to Endpoint Sentinel X! To maintain code quality and architectural integrity, please follow these guidelines.

## Branch Strategy

- **`main`**: The stable branch. Directly matches the latest production release.
- **`dev`**: The active development branch. Feature branches must branch off of and merge back into `dev`.
- **Feature Branches**: Named as `feature/sprint-<sprint-number>-<feature-description>` (e.g., `feature/sprint-1-jwt-auth`).

---

## Commit Message Convention

We follow conventional commits:

- **`feat:`**: A new feature (e.g., `feat: implement user creation API`).
- **`fix:`**: A bug fix (e.g., `fix: resolve jwt token expiration check`).
- **`refactor:`**: Code change that neither fixes a bug nor adds a feature.
- **`docs:`**: Documentation changes.
- **`chore:`**: Tasks, dependencies, build configurations.

---

## Pull Request Workflow

1. Branch off the `dev` branch.
2. Implement changes, adding appropriate unit/integration tests.
3. Verify formatting and linting pass (`ruff check` and `ruff format`).
4. Submit a Pull Request (PR) to merge into `dev`.
5. Ensure the CI pipeline passes.
6. A minimum of 1 code review approval is required before merging.

---

## Coding Standards

- **Python**: Follow PEP 8 styles, enforced strictly via Ruff (target version: Python 3.13+).
- **TypeScript**: Follow ES6+ guidelines, checked via ESLint.
- **SQLAlchemy**: Use SQLAlchemy 2.0 async paradigms exclusively.
- **Schemas**: Decouple database representations from client interfaces using Pydantic.

---

## Migration Policy

- **Never** manually alter production database schemas.
- Every schema change must be accompanied by an Alembic migration script.
- Ensure both `upgrade` and `downgrade` methods are implemented and tested.
- Review autogenerated migrations for unintended actions (e.g., drops of unrelated tables).

---

## Testing Requirements

- Maintain test coverage for all application logic in services and handlers.
- Mock external network resources (APIs, Redis, etc.) using pytest fixtures.
- Run `pytest` locally to confirm all tests pass before making a pull request.
